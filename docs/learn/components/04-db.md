# PostgreSQL 数据层

`packages/db` 是控制面的 PostgreSQL 契约层：它用 Goose migrations 定义持久模型，用 sqlc 生成类型安全查询，并为 core、Dashboard 与 Auth 暴露不同的 client 视图。

## 1. 系统位置

DB 包本身不是数据库服务，也不监听网络；它是多个 Go module 共同 import 的库与 migration 镜像。

```text
packages/api ----------- core queries -----\
packages/dashboard-api - dashboard/core ----> pgx pool -> PostgreSQL
packages/auth ---------- auth queries ------/
                              |
                     Goose migrations + sqlc
```

目录按职责拆分：

- `migrations/`：主 schema 的增量历史，也是生产 migrator 的输入。
- `queries/`：API 与 Dashboard 共用的资源查询及生成代码。
- `pkg/dashboard/sql_queries/`：Dashboard 页面投影查询。
- `pkg/auth/sql_queries/`：用户、identity、team 与凭证查询。
- `client/`、`pkg/auth/client.go`：连接池和查询集合的装配。
- `pkg/types/`：JSONB、build status 与 paused Sandbox config 等手写类型。

## 2. 启动/装配

主 migrator 是 [`packages/db/scripts/migrator.go`](../../../packages/db/scripts/migrator.go)。它用 `_migrations` 作为 Goose tracking table，获取 PostgreSQL session lock，再顺序应用 `packages/db/migrations`。

API 与 Dashboard API 启动时调用 `CheckMigrationVersion`，只接受“当前版本大于或等于构建期 expected migration”；它们不会在应用进程内自动执行 migration。

运行时连接由 `pool.New` 创建：

1. 解析 pgxpool DSN 与 caller options。
2. 安装 OpenTelemetry pgx tracer。
3. 使用 `QueryExecModeExec`，禁用 prepared statement cache。
4. 注册 `uuid[]` codec。
5. 创建 pool 并注册连接池 metrics。
6. 在返回给 sqlc 前套一层 transient-error retry wrapper。

[`packages/db/sqlc.yaml`](../../../packages/db/sqlc.yaml) 定义四次生成：core、Dashboard、Auth 和 testutils。Dashboard 的 sqlc schema 还读取 `pkg/dashboard/migrations`；主 migrator 只读取 `migrations/`，因此运行环境必须用部署流程确保 Dashboard 专属的 `env_defaults` 和 `teams.profile_picture_url` 已存在。

## 3. 核心机制与关键对象

| 对象 | 职责 | 特点 |
| --- | --- | --- |
| `client.Client` | core query facade | embeds `queries.Queries` |
| `client.Client.Dashboard` | Dashboard 专用查询 facade | 与 core 共用一个 pool/transaction |
| `authdb.Client` | Auth query facade | `Read` 与 `Write` 可绑定不同 pool |
| `pool.New` | 连接、telemetry、codec 与 retry 装配 | 返回 retry DBTX + raw pool |
| `retry.Wrapper` | 对可重试 pgx 错误执行退避重试 | 默认最多 5 次，带 jitter |
| `queries/*` | sqlc 生成的参数、row 与 method | 不手工编辑 |
| `pkg/types` | sqlc override 使用的领域值 | JSONB、BuildStatus、PausedSandboxConfig |

核心持久模型按关系可这样理解：

```text
public.users --< user_identities
      |
      +--< users_teams >-- teams -- tiers/addons -> team_limits view
      |                         |
      +--< access_tokens        +--< team_api_keys
                                +--< envs -> active_envs view
                                      |--< env_aliases
                                      |--< env_build_assignments >-- env_builds
                                      |--< snapshots
                                      |-- snapshot_templates
                                +--< volumes
                                +--> clusters
```

`active_template_builds` 是并发 build 配额的显式登记表；它和 `env_builds.status_group` 的索引共同支撑“活跃构建”查询，不应通过扫描全部历史 build 重新推断。

## 4. 主请求或数据流

### 普通查询

```text
handler/service
  -> sqlc method(typed params)
  -> retry DBTX
  -> pgxpool
  -> PostgreSQL
  -> generated Row / domain override type
```

retry 只处理被判定为 transient 的 DB 错误，并遵守 request context；业务约束、not found 和唯一键冲突仍原样返回，由上层映射为 HTTP/gRPC 状态。

### Core + Dashboard 共用事务

```text
db.Client.WithTx(ctx)
  -> pool.BeginTx
  -> core Queries.WithTx(tx)
  -> Dashboard Queries.WithTx(tx)
  -> caller 执行多条写入/锁定
  -> caller 显式 Commit 或 Rollback
```

`WithTx` 不自动提交。调用方通常立即 `defer tx.Rollback`，成功路径再显式 `Commit`。

### Auth 读写分离

```text
authdb.NewClient(primary, replica)
  -> Write -> primary pool
  -> Read  -> replica pool
       replica URL 为空时复用 primary

普通认证读取 -> Read
API key last_used / team mutation -> Write
刚 bootstrap 的 OIDC identity 查询 -> Write
```

这条边界是显式一致性选择：不能机械地把所有 SELECT 都切到 `Read`。

## 5. 设计不变量与故障边界

- schema 的事实来源是有序 migration，而不是生成的 `models.go` 或文档表格。
- SQL 源文件是查询事实来源；`*.sql.go` 是生成物，修改后会被下一次 `sqlc generate` 覆盖。
- `active_envs` 是 env 读路径的规范入口，用来排除 `deleted_at` 已设置的模板。
- Template 与 Build 是多对多关系，关联和 tag 位于 `env_build_assignments`，不能只依赖 `env_builds.env_id`。
- build 读侧分支应使用 `status_group`；原始 `status` 保存执行协议的细粒度历史值。
- team 限额来自 `team_limits` view，它合并 tier 基础值与有效 addons，不等同于只读 `tiers`。
- API key 与 access token 表保存 hash 和 mask metadata，不保存可再次读取的明文 secret。
- `user_identities` 以 `(oidc_iss, oidc_sub)` 为主键，并外键指向 canonical `public.users`。
- running Sandbox 不在 PostgreSQL 建立主记录；`snapshots` 保存 pause 后的 durable 状态，`billing.sandbox_logs` 保存历史 record。
- Auth read replica 允许短暂滞后；需要 read-after-write 的身份与管理路径必须使用 `Write` queries。
- pool retry 不等于事务重试；事务中出现 serialization/deadlock 后，caller 仍需决定是否重放整个事务。
- migration version check 只检测下限，不验证 schema 中每个对象；专属 migration 缺失仍可能在首个查询时报 undefined table/column。
- Dashboard 专属 migration 与主 migrator 目录分离；`sqlc` 能生成成功不证明部署数据库已应用该对象。

## 6. 与其他组件边界

- 与 `packages/api`：DB 持久化 template/build/snapshot/volume/cluster；Redis 中的 running Sandbox 与路由不属于本包。
- 与 `packages/dashboard-api`：Dashboard 使用 core queries 展示 build/record，使用 `Client.Dashboard` 展示 team/template 投影。
- 与 `packages/auth`：DB 提供 credential、identity、membership 与 team-limit 查询；凭证格式、OIDC 校验和缓存属于 auth。
- 与 ClickHouse：Sandbox metrics 与 team metrics 不在 PostgreSQL schema 中。
- 与 Loki：运行日志不由本包持久化；DB 只保存定位日志所需的资源 metadata。
- 与 migration 部署：应用只验证版本并使用 schema，真正的变更执行属于 migrator/job。
